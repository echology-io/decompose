Security Classification Lands
in Decompose v0.1.1

February 16, 2026
2 min read

Here's a question that comes up in every compliance review: which parts of this document contain security-relevant content?

Not "important" content. Not "high-priority" content. Security content. The sections that describe authentication flows, credential handling, access control, data exposure risks. The parts your security team needs to read before anything ships.

Most teams solve this one of two ways. They assign a human to read every document manually and tag the relevant sections. Or they send the whole document to an LLM and ask it to classify security risks. The first approach doesn't scale. The second gives you different answers on Tuesday than it gave you on Monday.

Decompose Now Classifies Security Risk

Starting with v0.1.1, Decompose ships a security risk category in its classification output. Every semantic unit gets scored. Deterministically. No model inference, no API call, no prompt engineering.

from decompose import decompose_text result = decompose_text(""" All API tokens must be rotated every 90 days. Tokens that exceed this period will be automatically revoked. Service accounts using static credentials are exempt from rotation but require quarterly access reviews. """) for unit in result["units"]: print(f"[risk={unit['risk_score']:.2f}] [{unit['category']}] {unit['text'][:80]}")

The output is the same every time you run it. Same input, same units, same scores, same classifications. No temperature knob. No stochastic variation. The security category now sits alongside the existing risk categories, giving you a structured way to filter, route, and prioritize document sections before any LLM touches them.

What This Looks Like in Practice

Feed Decompose a 40-page security policy. It splits the document into semantic units, classifies each one, and flags the sections that deal with authentication, authorization, credential management, and data exposure. Your downstream system — whether that's Claude, GPT-4, or a human reviewer — gets a pre-filtered, pre-classified feed instead of a raw document dump.

security_units = [u for u in result["units"] if u.get("category") == "security"] print(f"Found {len(security_units)} security-relevant sections")

This is mechanical work. Pattern matching against known structures. Decompose handles it in single-digit milliseconds so your LLM budget goes toward the parts that actually require reasoning.

What It Cannot Do

Decompose doesn't understand whether a security policy is *good*. It can tell you that a section describes credential rotation. It cannot tell you whether 90-day rotation is sufficient for your threat model, whether the exemption for service accounts creates an unacceptable risk, or whether the quarterly review cadence meets SOC 2 requirements.

That's domain reasoning. You need an LLM or a human for that. The point is that Decompose strips away the mechanical classification work so the expensive, slow, intelligent system can focus on judgment calls instead of parsing.

Why Deterministic Classification Matters for Security

If your AI system processes security documents and someone asks "why did it flag this section?" — you need a reproducible answer. "The model thought it looked security-relevant" is not an audit trail. A deterministic classifier that produces identical output on identical input is.

This is the difference between a system you can debug and a system you can only re-run and hope. In regulated industries — construction, healthcare, finance — document classification decisions carry liability. The classification layer should be the part of your stack you never have to wonder about.
