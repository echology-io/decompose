Classificação de Segurança
Chega ao Decompose v0.1.1

Uma categoria de risco que faltava — e por que seu pipeline precisa dela

16 de fevereiro de 2026
3 min de leitura

---

Você recebe um documento. Pode ser um contrato, uma especificação técnica, um relatório de engenharia. Seu pipeline precisa decidir: este texto contém informação sensível? Credenciais expostas? Dados pessoais? Cláusulas de confidencialidade?

Com um LLM, você manda o texto inteiro para uma API, espera 2-4 segundos, paga por token, e recebe uma resposta que pode mudar se você rodar de novo amanhã. Em um ambiente regulado — saúde, construção civil, financeiro — essa inconsistência não é um inconveniente. É um risco operacional.

O Decompose sempre classificou texto em unidades semânticas de forma determinística. Mesma entrada, mesma saída, toda vez. Sem LLM, sem chave de API, sem GPU. Mas até a versão 0.1.0, a classificação de risco cobria categorias como legal, financial, technical — e ignorava security.

Isso mudou na v0.1.1.

O que mudou

A categoria security agora faz parte da saída de classificação. Quando o Decompose decompõe um texto, cada unidade semântica recebe um score de risco. A partir da v0.1.1, riscos de segurança — credenciais expostas, referências a vulnerabilidades, dados pessoais, cláusulas de confidencialidade — são detectados e classificados como parte do pipeline determinístico.

Uma chamada de função. Sem modelo treinado. Sem dependência de nuvem.

from decompose import decompose_text text = """ Acesso ao servidor de produção: Host: 10.0.0.42 Usuário: admin Senha: s3nh@_pr0duc40_2024 Porta SSH: 22 """ result = decompose_text(text) for unit in result: print(f"[{unit['classification']}] risco={unit['risk_score']} → {unit['text'][:60]}")

A saída classifica cada unidade com sua categoria de risco. Credenciais são sinalizadas como security. Determinístico. Reproduzível. Sem latência de rede.

              Como funciona a classificação

              O Decompose não usa embeddings. Não usa modelo de linguagem. A classificação é baseada em análise determinística do texto — padrões léxicos, estrutura sintática, heurísticas calibradas. O resultado é um JSON estruturado com unidades semânticas, cada uma com:

              • `classification`: categoria semântica da unidade
                • `risk_score`: score numérico de risco
                  • `authority_level`: nível de autoridade da declaração
                    • `entities`: entidades extraídas
                      • `irreducible`: flag de irredutibilidade

                        • A categoria security se encaixa nesse framework existente. Não é um módulo separado — é uma expansão do vocabulário de classificação.

                        Configuração MCP

                        Se você usa Claude Code, Cursor, ou qualquer cliente MCP, a configuração é direta:

                        { "mcpServers": { "decompose": { "command": "uvx", "args": ["decompose-mcp", "--serve"] } } }

                        O agente ganha acesso a decompose_text e decompose_url como ferramentas. A classificação de segurança funciona automaticamente — não precisa habilitar nada.

                        O que o Decompose não faz

                        Clareza sobre limitações:

                        • **Não é um scanner de vulnerabilidades.** Ele classifica texto — não executa código, não faz análise estática, não testa endpoints.
                          • **Não substitui um DLP.** Se você precisa de Data Loss Prevention em escala empresarial com políticas granulares, o Decompose não é isso.
                            • **Não é probabilístico.** Isso é uma vantagem e uma limitação. Ele não vai "descobrir" padrões novos que não estão nas heurísticas. Mas também não vai inventar falsos positivos diferentes a cada execução.
                              • **Não processa imagens, áudio ou binários.** Texto. HTML. Markdown. Esse é o escopo.

                                • Por que isso importa

                                Em pipelines de dados que processam documentos de construção civil, contratos, especificações técnicas — a pergunta "este documento contém informação sensível?" aparece antes de qualquer outra decisão de roteamento.

                                Se a resposta depende de um LLM, você tem três problemas: custo por documento, latência por chamada, e não-determinismo entre execuções. Em um audit trail, "o modelo disse que era seguro" não é uma resposta aceitável. "A classificação determinística retornou risk_score=0 para security em todas as 14 unidades do documento" — isso é auditável.

                                O Decompose não compete com LLMs. Ele cobre a camada que vem antes: triagem determinística, classificação reproduzível, extração estruturada. O LLM entra depois, nos casos que realmente precisam de raciocínio. O resto — e na maioria dos pipelines, "o resto" é 80-90% do volume — se resolve com uma chamada de função que roda em milissegundos.
